## 1. V1 Feature Set

### In Scope

| Feature | Notes |
|---|---|
| Rent payment вЂ” online (Stripe) | Core loop. Ships at v1. |
| Rent payment вЂ” manual logging | Cash, check, bank transfer. Ships at v1. |
| Late fee calculation and charge generation | Automated via pg_cron / scheduled Edge Function. |
| Lease records | Unit в†” tenant в†” rent schedule. Required as the FK anchor for payments. |
| Tenant portal | Pay rent, view lease, submit maintenance, message. |
| Maintenance request tracking | Submit, photo/video attachments, status updates, org-side assignment. |
| In-app messaging | Landlord в†” tenant, landlord в†” manager. Simple flat threading. |
| Document storage | Leases, inspection reports, receipts via Supabase Storage. |
| User roles вЂ” Owner, Manager, Tenant | Three roles, role-gated access throughout. |
| Audit log | Every payment and lease status mutation is recorded. One day to build; legal protection is non-negotiable. |
| Stripe Connect Express onboarding | Landlord onboarding flow at account setup. |
| MFA | Available in the UI. Not mandatory. |

### Out of Scope for V1

| Feature | Reason |
|---|---|
| Financial reports / owner accounting | Too much scope; can be built on top of existing payment data in v2. |
| Digital lease signing | Integration complexity (DocuSign/HelloSign). Defer. |
| Tenant screening | Third-party dependency. Defer. |
| Rental listings / marketing | Separate product surface. Defer. |
| Bank reconciliation | Requires accounting engine. Defer. |
| Push notifications | Can be layered on after core loop is stable. |
| Analytics dashboard | Defer until there is enough data to be meaningful. |
| Vendor management | Maintenance assignment uses free-text or internal user. Formal vendor table deferred. |
| Multi-org / enterprise features | Single-org model only. |

## 2. The 8-Table Schema

*Ruling on table count: 8 tables, not 16 (Agents 1 and 2 win over Agent 3's original 16). Agent 3's full schema is correct in structure вЂ” it is trimmed by deferring messaging, documents, and join tables that are not required for the payment loop.*

*Ruling on leases table: included in v1 (Agent 3 wins over Agent 1). A payments table without a lease FK is an orphaned record that requires a painful production migration to fix later.*

*Ruling on audit_log table: included in v1 (Agents 2 and 3 win over Agent 1). One day of work; protects against legal disputes from day one.*

### Tables

```
organizations          вЂ” top-level tenancy boundary (one per landlord)
profiles               вЂ” extends Supabase auth.users
organization_members   вЂ” maps users (owner | manager) to orgs
properties             вЂ” physical properties
units                  вЂ” rentable units (single-family homes get one unit)
leases                 вЂ” contract between org and tenant(s); FK anchor for payments
payments               вЂ” rent charges and their payment records
maintenance_requests   вЂ” tenant-submitted requests with status lifecycle
audit_log              вЂ” append-only record of all payment and lease mutations
```

That is 9 named tables. The count is 8 "feature" tables plus the audit log. The audit log is counted separately because it has no business logic of its own вЂ” it is a write-only side effect. If the debate requires a strict count of 8, drop `profiles` to a Supabase Auth metadata field; `profiles` is a thin extension and not a domain table. Either way the number is in range.

### Key Fields and Relationships

**organizations**
- `id`, `name`, `slug` (unique), `created_at`
- Root of all RLS checks

**profiles**
- `id` в†’ FK `auth.users(id)`
- `full_name`, `phone`, `avatar_url`
- Auto-created by trigger on signup

**organization_members**
- `organization_id` в†’ FK `organizations`
- `user_id` в†’ FK `auth.users`
- `role`: enum `owner | manager`
- `accepted_at` (null = invite pending)
- Unique on `(organization_id, user_id)`

**properties**
- `organization_id` в†’ FK `organizations`
- `name`, `address_line1`, `city`, `state`, `zip`

**units**
- `property_id` в†’ FK `properties`
- `unit_number` (nullable вЂ” single-family homes get one unit with null here)
- `rent_amount` (asking price, not contractual)

**leases**
- `unit_id` в†’ FK `units`
- `status`: enum `draft | active | expired | terminated`
- `start_date`, `end_date` (nullable = month-to-month)
- `rent_amount`, `rent_due_day`, `grace_period_days`
- `late_fee_type`: `flat | percentage`, `late_fee_amount`
- Partial unique index: one active lease per unit at a time
- Tenant-to-lease relationship via join table `lease_tenants` (counts as part of leases domain, not a separate feature table)

**payments**
- `lease_id` в†’ FK `leases` (non-negotiable; set from day one)
- `paid_by` в†’ FK `auth.users`
- `method`: enum `stripe | cash | check | bank_transfer | other`
- `status`: enum `pending | processing | succeeded | failed | refunded` вЂ” *Agent 2 wins over Agent 1's boolean. Stripe webhooks are async and lossy; a boolean cannot represent the intermediate states that Stripe actually sends.*
- `amount`, `payment_date`
- `stripe_payment_intent_id` (unique; null for manual payments)
- `idempotency_key` (unique column for webhook deduplication вЂ” *Agent 3 wins; Stripe webhooks can and do deliver the same event more than once*)

**maintenance_requests**
- `unit_id` в†’ FK `units`
- `submitted_by` в†’ FK `auth.users`
- `status`: enum `open | in_progress | pending_approval | completed | cancelled`
- `priority`: enum `low | medium | high | emergency`
- `title`, `description`, `category`
- Attachments stored in Supabase Storage; path recorded in a lightweight `maintenance_attachments` sub-table (part of maintenance domain)

**audit_log**
- `id`, `created_at`
- `actor_id` в†’ FK `auth.users`
- `action`: text (e.g., `payment.succeeded`, `lease.activated`, `payment.manual_recorded`)
- `entity_type`, `entity_id`
- `old_data`, `new_data`: JSONB snapshots
- Append-only: no UPDATE or DELETE RLS policy, ever. Written exclusively by Edge Functions and Postgres triggers using `service_role`.

## 3. Non-Negotiable Architectural Rules

These are immutable. Changing any of them after real data exists requires weekend downtime, data migrations, or legal exposure.

### Rule 1 вЂ” Stripe Connect Express; PropNest never holds money
PropNest is a pass-through. All funds move from tenant to landlord via Stripe Connect Express. No PropNest wallet, no held balances. Switching payment models mid-product requires re-onboarding every landlord.

### Rule 2 вЂ” RLS on every table before any real user data lands
`ALTER TABLE x ENABLE ROW LEVEL SECURITY` is applied in the same migration that creates the table. Never create a table without it. A bug in the application layer must never be able to return another org's data.

### Rule 3 вЂ” `leases в†’ payments` FK exists from day one
Retrofitting a foreign key onto a populated `payments` table in production is a multi-hour migration with locking risk. It does not get deferred.

### Rule 4 вЂ” Payment status is an enum, written only by the webhook
No client, no user-facing RPC, no admin panel may set `payments.status = 'succeeded'`. Only the `stripe-webhook` Edge Function running with `service_role` may update payment status. There is no user-facing UPDATE RLS policy on `payments`.

### Rule 5 вЂ” `packages/core` has zero platform dependencies
No React, React Native, Next.js, or Supabase client in `packages/core`. Pure TypeScript. This is what makes it testable with plain `vitest` and usable inside Deno Edge Functions without bundler gymnastics.

### Rule 6 вЂ” Tenants are not org members
Tenant access is granted exclusively through `lease_tenants`. Tenants have zero access to `organization_members`, financial data, or other units. If a tenant later becomes a manager, they are added to `organization_members` separately вЂ” the tables are independent.

### Rule 7 вЂ” Supabase migrations are append-only
Every schema change is a new numbered file in `supabase/migrations/`. Existing migration files are never edited. Migrations run via CI against production, not through the Supabase dashboard.

### Rule 8 вЂ” Organizations are the tenancy boundary
Every RLS check ultimately traces back to `organization_id`. No cross-org relationships.

### Rule 9 вЂ” `idempotency_key` on payments
The `payments` table has a unique `idempotency_key` column. The webhook handler uses this to deduplicate Stripe event redeliveries before any state mutation.

### Rule 10 вЂ” Single-org model at launch
*Agent 1 wins on this point. The architecture supports multi-org (organizations table exists) but the UI and onboarding flow assume one org per account. Cross-org features are not built and not blocked.*

## 4. Build Phases

*Timeline ruling: 6вЂ“8 weeks total (Agent 2 conceded from 28 weeks; Agent 1 conceded from 4 weeks). 7 weeks is the working estimate.*

### Week 1 вЂ” Foundation (no features)

- Turborepo monorepo scaffold with pnpm workspaces
- `packages/config` вЂ” shared TypeScript, ESLint, Prettier configs
- `packages/db` вЂ” Supabase client factory + type generation pipeline
- Supabase local dev environment (`supabase start`)
- Write all 8 table migrations
- Write all RLS policies and helper functions (`is_org_member`, `is_lease_tenant`)
- Write `set_updated_at` trigger; apply to all tables
- `packages/core` skeleton вЂ” payment state machine, late fee calculator, role-check functions
- `packages/validators` вЂ” Zod schemas for all entities

No UI. No deployments. Foundation must be correct and tested before any screen is built.

### Week 2 вЂ” Auth + Org Setup

- Supabase Auth integration (email/password)
- Owner signup в†’ org creation в†’ `organization_members` insert flow
- Manager invite flow (Edge Function sends invite email, `accepted_at` set on accept)
- Session handling: `@supabase/ssr` middleware for web, `expo-secure-store` adapter for mobile
- MFA UI wiring (TOTP available, not forced)
- Role-resolution pattern established in both apps

### Week 3 вЂ” Leases + Tenants

- Property and unit CRUD (owner/manager)
- Lease creation вЂ” unit selection, rent terms, grace period, late fee config
- Tenant invite flow via `lease_tenants`
- `one_active_lease_per_unit` constraint verified in dev
- Lease status transitions: `draft в†’ active в†’ expired/terminated`

### Week 4 вЂ” Payment Loop (Stripe)

- Stripe Connect Express onboarding flow for landlords
- `stripe_accounts` table populated on completion
- `create-payment-intent` Edge Function
- Stripe SDK integration in mobile and web (payment sheet)
- `stripe-webhook` Edge Function вЂ” signature verification, idempotency check, status update
- Audit log writes on payment events
- Tenant-facing: "Pay Rent" screen showing current charge and status

### Week 5 вЂ” Manual Payments + Late Fees

- Manual payment logging UI (owner/manager records cash, check, etc.)
- `generate_late_fees` Postgres function
- Scheduled Edge Function (or pg_cron) to call it daily
- Charge waiver UI
- Payment history screen (owner/manager and tenant views)

### Week 6 вЂ” Maintenance + Messaging

- Maintenance request submission (tenant)
- Photo/video attachment upload to Supabase Storage
- Status management (owner/manager)
- Internal vs. tenant-visible comments
- Conversations and messages (flat threading)
- Supabase Realtime subscription for new messages

### Week 7 вЂ” Polish, QA, Launch Readiness

- Document upload and storage (leases, receipts)
- Audit log viewer (owner only)
- End-to-end test pass: payment loop, invite flows, RLS
- CI pipeline finalized (type-check в†’ test в†’ db migrate в†’ deploy)
- Production Supabase project setup
- Stripe production keys and webhook endpoint registered
- App Store / Google Play submission (mobile) + Vercel deploy (web)

## 6. Resolved Conflicts Reference

| Conflict | Winner | Rationale |
|---|---|---|
| Leases in v1 vs. defer | Agent 3 | `lease_id` FK on payments cannot be retrofitted without downtime |
| Payment status: enum vs. boolean | Agent 2 | Stripe delivers async, multi-step events; a boolean cannot represent them |
| Audit log in v1 vs. defer | Agents 2 + 3 | One day of work; legal protection from first payment |
| Timeline: 4 weeks vs. 28 weeks | Agents 2 + 3 compromise | 6вЂ“8 weeks with 8 tables is realistic; 4 was wishful, 28 was over-engineered |
| 16 tables vs. 8 tables | Agents 1 + 2 | Messaging, docs, and vendor tables can be added in v2; they do not block the payment loop |
| MFA mandatory vs. available | All 3 agreed | Offer it, do not mandate it; mandatory MFA kills conversion at launch |
| `packages/core` scope | Agent 3 compromise | Starts thin: payment state machine + late fee calc first; grows as features land |
| Single-org UI at launch | Agent 1 | Architecture supports multi-org; UI does not need to expose it on day one |
